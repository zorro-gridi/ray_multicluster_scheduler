# 模块探索与架构理解规范

在进行项目代码探索、功能分析或重构建议时，必须遵循“文档先行”原则。

### 1. 探索流程强制要求
*   **严禁直接阅读源码**：在未阅读对应模块的文档前，禁止直接打开或读取该模块的核心逻辑代码。
*   **首要任务**：必须首先定位并阅读该模块目录下的 `docs/` 子目录（或项目约定的接口说明文档存放路径）。
*   **核心阅读目标**：
    *   **接口定义**：了解该模块对外暴露的 API 或服务协议。
    *   **基本架构**：明确模块内部的分层与设计模式。
    *   **上下游依赖**：识别该模块调用了哪些组件，以及被哪些组件所调用。

### 2. 执行步骤
1.  使用 `ls` 命令检查目标模块是否存在 `docs/` 或类似的文档目录。
2.  优先阅读该目录下的 `README.md`、`architecture.md` 或 `api_spec.md` 等文件。
3.  在对话中向用户简要概述你从文档中理解到的架构逻辑，确认理解无误后再深入源码。

### 3. 特殊情况处理
*   若对应模块目录下缺失 `docs/` 目录，请在继续操作前告知用户：“未发现模块文档，将尝试通过全局 README 或代码注释进行推断。”
